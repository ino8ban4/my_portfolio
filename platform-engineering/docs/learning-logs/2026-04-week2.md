## Issue #13: Helmマルチ環境管理（dev/prod）

### やったこと
- values-dev.yaml / values-prod.yamlを作成
- ArgoCDのApplicationをdev/prod別に作成
- dev(replica:1) / prod(replica:3)で別namespaceにデプロイ確認

### 理解の証明

#### 理解内容
- Helmは複数のvaluesファイルをマージして使える
- 後から指定したファイルのキーが優先される
- ArgoCDのApplicationをenv別に分けることでデプロイ先を制御できる

#### 判断基準
- CreateNamespace=true はdev/stagingはOK、prodは外すべき
- namespace分離によりリソース・権限・通信を環境ごとに制御できる

#### 実装での選択理由
- values.yamlに共通値、values-dev/prod.yamlに差分だけ書く構成にした理由：
  重複を避け、変更箇所を最小限にするため

## Issue #14: Mini IDP構築（Platform Engineering集大成）

### やったこと
- platform-engineering/README.mdを作成
- Mermaidでアーキテクチャ図を作成
- 設計思想・技術スタック・デプロイフローをドキュメント化

### 理解の証明

#### 理解内容
- IDPは開発者のセルフサービス化によりインフラチームの属人化・ボトルネックを解消する
- ArgoCDがGitをSingle Source of Truthとして管理し、k8sの状態をGitに強制同期する
- GitHub ActionsはCI（ビルド・push）、ArgoCDはCD（デプロイ）と責務が明確に分離されている

#### 判断基準
- git pushからデプロイまでのフローがシームレスであることがMini IDPの最大の価値
- 設計思想（なぜこの構成にしたか）を説明できることがポートフォリオの差別化ポイント

#### 実装での選択理由
- MermaidでアーキテクチャをREADMEに埋め込んだ理由：GitHubで直接レンダリングされるため、別途ツールなしに構成を可視化できるため
