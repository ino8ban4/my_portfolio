## Issue #15: セキュリティ強化（NetworkPolicy / 脆弱性スキャン）

### 実施日
2026-04-21 〜 2026-04-23

### 学習内容

#### NetworkPolicy
- KubernetesはデフォルトでPod間の通信が全許可のため、NetworkPolicyでホワイトリスト制御を導入
- Helmチャートに`networkpolicy.yaml`を追加し、dev環境に適用
- Ingress: port 3000のみ許可
- Egress: DNS（port 53）のみ許可

#### Trivy
- コンテナイメージの脆弱性スキャナー
- `aquasec/trivy` DockerイメージでローカルスキャンをDockerコンテナとして実行できる
- GitHub Actionsに`aquasecurity/trivy-action`を組み込み、CI実行時に自動スキャン
- 結果: HIGH 12件、CRITICAL 0件を検出（cross-spawn CVE-2024-21538など）
- 現在は`exit-code: '0'`で検知のみ、本番運用では`'1'`でCIをブロックする

### 理解の証明

**なぜNetworkPolicyの適用にCalicoが必要か**
NetworkPolicyはKubernetesのAPI定義に過ぎず、実際の通信制御はCNIプラグインが担う。デフォルトのCNIはNetworkPolicyを実装していないため、Calicoのような対応CNIが必要。`minikube --cni=calico`で起動済みのためそのまま利用できた。

**なぜEgressにDNS（port 53）を許可するか**
Egressを制限するとPod内からのドメイン名解決が不可能になる。DBホスト名や外部APIのURLを名前で引けなくなるため、DNS通信は最低限許可する必要がある。

**TrivyをローカルにインストールせずDockerで実行できる理由**
Trivyはスキャン対象のイメージをpullして検査するため、スキャナー自体もコンテナとして動かせる。`--rm`フラグで実行後にコンテナは自動削除されローカル環境をクリーンに保てる。

**CI組み込みにおけるexit-codeの意味**
`exit-code: '1'`にするとHIGH以上の脆弱性が検出された時点でCIが失敗しマージをブロックできる。現在は`'0'`で検知のみだが、チームのセキュリティポリシーに応じて切り替える設計判断が必要。

### トラブルシューティング
- `helm upgrade`失敗: 既存のSecretがHelmの管理外で作成されていたため、`kubectl label/annotate`でHelmに引き取らせた
- ArgoCD全Pod Error: `kubectl rollout restart`で復旧

## Issue #17: 監視基盤構築（Prometheus / Grafana）

### 実施日
2026-04-26

### 学習内容
- kube-prometheus-stackをHelmでインストール
- Grafanaにport-forwardでアクセスし、ダッシュボードを確認
- CPU・メモリ等の基本メトリクスをGrafana上で可視化

### 理解の証明

**PrometheusとGrafanaの役割の違い**
Prometheusはメトリクスを収集・保存する役割、GrafanaはそのデータをUIで可視化する役割。データの蓄積と表示が分離されている。

**kube-prometheus-stackを使う理由**
Prometheus・Grafana・Alertmanager・各種Exporterを個別に設定すると連携設定が煩雑。kube-prometheus-stackはK8s向けの設定済みで一括インストールできる。ダッシュボードもプリインストール済み。

**node-exporterとkube-state-metricsの違い**
node-exporterはNodeのハードウェア情報（CPU・メモリ・ディスク）を取得し各Nodeに1つずつ動く。kube-state-metricsはK8sリソースのState（PodのStatus・Deploymentのreplica数等）をAPIから取得する。
